# em walk

### Purpose

Traverse forward, backward, or both from any node and return a bounded local subgraph.

Use this when you know a starting component but need a **multi-hop exploration command** that does not require loading the whole graph.

### Syntax

```bash
em walk --from <nodeId|canonicalId> [--direction forward|backward|both] [--edge-type <type> ...] [--max-hops N] [--limit N] [--cursor <cursorId>]
```

### Semantics

- `walk` is a **bounded traversal command**.
- It is optimized for iterative exploration in large graphs.
- It returns a **path-oriented local subgraph**, not disconnected node and edge lists.
- It should stop at `maxHops` or `limit`, then return `frontier.nextCursor` if more traversal remains.
- Typical use:
    - backward from a UI component to find view → event → command
    - forward from a command to find event → view → processor

### Traversal semantics

#### `--direction backward`

- Start from `--from` as the root.
- At each step, follow edges that point **into the current node**.
- Output is still written from the root outward as a readable branch, but each traversed edge is marked with `"direction": "backward"`.
- Typical meaning:
    - from UI/component backward = trace what data or commands explain the current UI
    - from viewModel backward = find upstream events and commands
    - from event backward = find producer commands

#### `--direction forward`

- Start from `--from` as the root.
- At each step, follow edges that point **out of the current node**.
- Output is written from the root outward and each traversed edge is marked with `"direction": "forward"`.
- Typical meaning:
    - from command forward = inspect downstream impact
    - from event forward = inspect refreshed views, updated processors, and effects
    - from viewModel forward = inspect consumers

#### `--direction both`

- Combine both forward and backward expansion around the same root.
- The result is still bounded by `maxHops` and `limit`.
- Use this when the starting node sits in the middle of a workflow, especially for events and viewModels.
- To keep the result readable, `both` mode should **not** mix all branches into one list.
- Instead, it should return:
    - `subgraph.backwardBranches[]`
    - `subgraph.forwardBranches[]`

#### Output rules

- The output always keeps a single `subgraph.root`.
- For `forward` or `backward`, the output returns `branches[]`.
- For `both`, the output returns two separated lists:
    - `backwardBranches[]`
    - `forwardBranches[]`
- Every branch is a readable alternating path of:
    - node
    - edge
    - node
    - edge
    - node
- Branches may share prefixes.
- The command should deduplicate already-visited node/edge pairs within the same walk call.
- If traversal stops because of `limit` or `maxHops`, the remaining frontier is represented by `frontier.hasMore` and `frontier.nextCursor`.

### LLM usage guidance

- Use `em walk` when one-hop neighbors are not enough.
- Prefer directional exploration:
    - from UI backward to understand why the UI changes
    - from command forward to understand impact
    - from event both ways to inspect producers and consumers
- Prefer `both` mainly for middle-layer nodes like `evt` and `viewModel`.
- Keep hops small at first. Expand only as needed.
- If `frontier.hasMore` is true, continue with `--cursor` instead of restarting from scratch.
